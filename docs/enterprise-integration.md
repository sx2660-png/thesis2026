# Trading Brain 每日风险简报：企业对接说明

本文档说明每日风险简报如何被企业系统调用、哪些字段可以展示给交易员，以及失败和反馈如何处理。当前默认场景是广东日前市场，系统提供的是外部信息风险分析，不是申报、策略或审批指令。

## 示例报告

企业可以先查看这份[每日风险简报示例：2026-05-23](reports/daily_risk_brief_2026-05-23.md)，了解交易员摘要、预测上下文、证据、数据质量和技术审计信息在实际输出中的组织方式。

## 1. 对接目的与边界

Trading Brain 用于判断：外部新闻、公告、政策背景和最新预测基线，是否可能使现有预测基线失准。它输出每日分析和关注程度，帮助交易员决定是否需要进一步查看。

默认分析截点为目标交易日 D-1 的 10:00（Asia/Shanghai）。系统只应使用该截点前可获得的信息。Trading Brain 不直接替代企业的预测、策略、申报或审批系统。

## 2. 在线调用

接口：`POST /workflow/run`

请求示例：

```json
{
  "trade_date": "2026-05-23",
  "as_of_datetime": "2026-05-22T10:00:00+08:00"
}
```

响应外层包含 `trace_id`、`trade_date`、`status` 和 `daily_risk_brief`。企业前端默认只展示 `daily_risk_brief` 中面向交易员的字段；`audit` 供技术排查和离线评估使用，默认隐藏。

## 3. 每日风险简报字段

| 字段 | 类型 | 对接含义 |
| --- | --- | --- |
| `trade_date` | string | 目标交易日，格式为 `YYYY-MM-DD`。 |
| `analysis_cutoff` | ISO-8601 datetime | 本次分析允许使用的信息截点。 |
| `status` | enum | `正常`、`关注`、`重点关注`、`信息不足`。 |
| `summary` | string | 给交易员阅读的简明分析，不包含操作指令。 |
| `risk_factors` | string[] | 可能涉及供需、价差、阻塞/检修、规则参数等因素。 |
| `baseline_impact` | string[] | 可能影响负荷预测、电价预测，或同时影响两者。 |
| `affected_periods` | object[] | 可能受影响的时段；无法判断时标记为待确认。 |
| `forecast_context` | object | Energyun 最新预测、可用性和版本比较状态。 |
| `evidence` | object[] | 可以展示给交易员的新闻或公告证据。 |
| `data_quality` | object | 新闻和预测基线是否可用。 |
| `audit` | object | 去重、技术评分、原始检索结果和 legacy 信息，默认隐藏。 |

### 状态解释

- `正常`：没有发现当前且相关的外部证据。
- `关注`：发现单条相关信息，或发布时间/事件时间仍待确认。
- `重点关注`：发现时新的官方公告，或多条独立来源共同支持同一风险因素。
- `信息不足`：搜索或预测基线不可用，表示系统不能判断，不能被展示为市场异常。

## 4. Energyun 预测基线

当前通过 Energyun 的价格预测和统一调度/负荷图表读取最新可用预测序列。`forecast_context.available` 表示至少价格预测基线可用；如果负荷序列获取失败，失败原因会保留在 `forecast_context.note`，不会覆盖价格基线的可用状态。

D-4、D-3、D-2、D-1 的版本比较，需要企业提供历史模型版本接口或数据导出。版本变化只用于解释预测上下文的差异，不能单独把状态升级为 `重点关注`。

## 5. 新闻来源与时间

来源优先级如下：

1. 官方和监管来源，例如交易中心、电网、监管机构和政府公告；
2. 行业和财经媒体，作为补充证据。

每条 `evidence` 应包含 `source_tier`、`published_at`、`event_time`、`time_verified` 和链接。转载内容在审计层去重，不能因为文章数量增加就提高风险状态。没有可靠时间的材料只能进入较低确定性的 `关注`，不能单独升级为 `重点关注`。

## 6. 交易员反馈与历史

交易员可以通过以下接口保存每日标签：

`PUT /alerts/feedback`

```json
{
  "trade_date": "2026-05-23",
  "worth_alerting": true,
  "risk_factors": ["价差", "价格结构"],
  "reason": "外部信息与预测偏离判断值得保留"
}
```

每个交易日保留一条可更新的反馈记录。`GET /alerts/history` 返回系统简报和交易员手动标注的合并结果，可用于“历史提醒与反馈”页面以及后续评估。

## 7. 失败与重试

| 情况 | 系统输出 | 企业侧处理 |
| --- | --- | --- |
| Tavily 搜索失败 | `status=信息不足`，`news_available=false` | 显示“无法判断”，记录运行错误并重试。 |
| Energyun 基线失败 | `status=信息不足`，`forecast_baseline_available=false` | 检查网络、token 和目标日期数据是否可用。 |
| 只有预测版本变化 | 保持原有新闻状态 | 只在预测上下文展示版本变化。 |
| 重复转载 | 在审计层去重 | 不根据文章数量提升风险状态。 |

当系统返回 `信息不足` 时，企业前端应明确告诉交易员“当前无法判断”，而不是将其显示为异常、风险或高等级告警。

## 8. 非默认 legacy 字段

旧的策略、自动执行、暂停提交、人工复核，以及 `alert_level`、`review_level`、`risk_status` 字段，仅为未来企业实验集成保留。默认的每日风险简报和交易员页面不依赖这些字段。

## 9. 前端展示建议

企业前端可以优先展示目标交易日、分析截点、当前状态、摘要、风险因素、影响时段、预测上下文和证据列表。技术审计信息、原始检索结果和内部评分默认放在展开层或技术页面中，避免让交易员把系统状态误解为交易指令。
